# Mories 4D Cognitive Engine - Handover Document
**생성일:** 2026-04-04
**프로젝트:** mirofish-supermemory

## 1. 현재 작업 상태 및 달성 현황
* **완료된 작업:**
  * Mories 시스템의 과거 기억 참조 및 자가 진화 파이프라인(4D Cognitive Engine) 구축 완료.
  * `Time-Travel Chat` (`GET /api/memory/<uuid>/chat?revision_id=...`) 구현 및 검증 완료.
  * `Hindsight Synthesis` (`POST /api/memory/<uuid>/synthesize`) 기능을 통해 과거 기억으로부터 새로운 통찰(Synthesis) 노드 생성 및 `SYNTHESIZED_FROM` 관계 맵핑 완료.
  * `MemoryManager`의 STM -> LTM 승격 라이프사이클을 포함하여 통합 테스트 32건(TDD) 모두 무결성 검증 통과 완료.
* **다음 목표(진행할 작업):** 
  * "멀티 에이전트 자율 실험 시나리오" 적용 (예: PINN 제품 수명 최적화 시나리오 또는 서브 에이전트 간 소설 창작 협업).
  * 외부 에이전트가 Mories 그래프에 접근하여 조직 메모리(LTM)와 개인 메모리(STM)를 오가며 자율적으로 협업하는 아키텍처 실증.

## 2. 엄격한 코딩 및 개발 지침 (차기 세션에서 반드시 준수할 것)
1. **가상환경 필수 사용:** Python 코드를 실행하거나 의존성을 확인할 때 시스템 글로벌이 아닌 프로젝트의 `.venv` 또는 `venv` 환경을 반드시 활성화시키고 진행할 것.
2. **테스트 파일 격리:** 추가로 생성할 테스트용 스크립트나 목업(Mockup) 모델 등은 메인 로직에 섞지 않고, 나중에 `rm` 명령어로 쉽게 일괄 지울 수 있게 `test_reports/`나 최상단 `tmp_tests/` 같은 일회성 전용 분리형 디렉터리에 저장할 것.
3. **공용 자원 수정 금지:** 다른 프로젝트와 얽혀있는 공용 자원(예: 전역 `docker-compose.yml`, 비밀번호/API 키 하드코딩 등)의 임의 수정을 절대 금지함.
4. **파일 크기 조절 (코드 분할):** 한 파일의 라인 수가 너무 길어지지 않게 단일 책임 원칙(SRP)에 기반해 모듈을 분리할 것.
5. **안정성 (예외 처리 & 로깅):** 에이전트가 자원을 점유하거나 API를 호출할 때 네트워크/권한/타임아웃 이슈가 발생할 수 있으므로 항상 `try-except-finally`를 사용하여 자원 미반환 오류를 예방하고 명시적인 로깅 처리를 추가할 것.
6. **주석 정책 (한글):** 코드의 동작 원리는 주석으로 한글로 최대한 상세히 남겨 개발자가 쉽게 유지보수할 수 있게 할 것. 복잡도가 높으면 모듈별로 별도의 설명서(MD 문서)를 생성하고 소스 코드 최상단 주석 영역에 해당 설명서의 1) 경로 2) 파일명 3) 버전을 명시할 것.
7. **트러블슈팅 이력 추적:** 동일한 이슈를 겪지 않기 위해 에러 수정 과정 및 원인을 `.agent/troubleshooting_log.md` 같은 문서에 누적 기록하고 주기적으로 참고할 것.
8. **컨텍스트 길이 모니터링:** 대화나 컨텍스트 창이 지나치게 길어져 LLM 효율이 저하되기 전에, 새로운 대화창으로 넘어갈 것을 사용자에게 먼저 제안할 것.

## 3. 남은 이슈 및 참고
* Mories MCP 서버와 `mirofish-supermemory`를 활용해 외부 클라이언트(예: Cursor, Claude, n8n)가 연결하는 부분의 오케스트레이션이 다음 시나리오의 핵심입니다. Webhook 설정 연동 여부 및 테스트 스텁(Stub) 처리에 주의하여 에러를 다스리세요.
